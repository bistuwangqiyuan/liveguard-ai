# 智能体可视化运行界面（Agent Runtime UI）开源项目调研报告

> 用途：为「在开源项目基础上自研一款商业级智能体可视化运行界面」提供技术选型与设计参考。
> 研究范围：以 [paperclipai/paperclip](https://github.com/paperclipai/paperclip) 为基准，覆盖通用 Agent 控制台、工作流编排、IDE 内嵌编程 Agent、多 Agent 协作、Agent 可观测性、Chat/Tool 渲染基础库 等六个赛道。
> 数据时点：2026-04-22；GitHub Star 数据来自 GitHub REST API（live）。

---

## 0. TL;DR — 给到决策者的一页结论

| 你的目标 | 推荐基座 | 理由 |
|---|---|---|
| 想要做"AI 员工/数字公司"控制台（组织架构 + 任务委派 + 成本 + 审计） | **Paperclip** + **assistant-ui** | 直接对标，MIT，TypeScript/React，自带 Org Chart、Ticket、Cost、Audit 四大主视图 |
| 想要做"通用智能体运行容器"（聊天 + 浏览器 + 文件 + 终端 + Artifact） | **Suna**（Kortix）或 **OpenHands** | Suna 商业 UI 最完整，Apache 2.0；OpenHands 70k+ Star、社区最活跃 |
| 想要做"低代码工作流式 Agent 编排" | **Dify** 或 **Langflow** | Dify 偏成品（Apache 2.0+附加条款），Langflow MIT 更纯净 |
| 想要做"多 Agent 协作可视化（看智能体怎么吵架/分工）" | **AutoGen Studio** + **AgentScope Studio** | 微软 + 阿里两个学术派代表，前者有 Team Builder + Playground，后者有完整 Trace |
| 只想做"前端 SDK/嵌入到自有产品" | **assistant-ui** + **tool-ui** | shadcn 风格、可拷贝可改、Tool Call 渲染最成熟 |
| 想要做 Agent 可观测/复盘（轨迹时间线 + Token + 成本 + 评测） | **Langfuse** + **Arize Phoenix** | Langfuse Web UI 工程化最强，Phoenix Trace 可视化最强 |

**最终建议（商业项目）**：
- **前端 UI 基座** = `assistant-ui`（消息流 + Tool Call 渲染） + `tool-ui`（25+ 受控组件）
- **运行编排参考** = `Suna` 的左右分栏「对话 / 沙箱视图」 + `Paperclip` 的「组织视图 + 成本看板 + 审计日志」
- **可观测层** = 内置 OpenTelemetry，兼容 `Langfuse` / `Phoenix`，避免重造轮子
- **License 红线**：避开 AGPL（如部分 LobeChat 历史版本/n8n fair-code），优先 MIT / Apache 2.0

## 1. 评估维度（你拿这套维度去看每一个项目都不会跑偏）

一个"商业级智能体可视化运行界面"必须同时回答以下 6 个问题：

1. **运行视图（Runtime View）**：用户能不能实时看到 Agent 现在在「想什么 / 调什么工具 / 看到什么 / 写了什么文件 / 浏览了哪个网页」？
2. **干预视图（Human-in-the-Loop）**：能不能在中途改主意、批准敏感动作、回滚到某一步、给出指令？
3. **编排视图（Orchestration）**：能不能编辑/调试 Agent 自身（提示词、工具、子 Agent）？是低代码画布还是代码？
4. **协作视图（Multi-Agent）**：多 Agent 之间的对话/分工/汇报关系如何呈现（Org Chart / Swimlane / 3D World）？
5. **观测视图（Observability）**：执行轨迹（Trace）、Token、成本、Token/秒、错误率、评测分数如何看？
6. **资产视图（Artifact）**：文件、图表、PDF、网页快照、代码 Diff 等中间产物如何沉淀与呈现？

下文每个项目按这 6 项打分（▲优秀 ●及格 ○缺失），并给出可借鉴的具体 UI 模式。

## 3. 赛道一：通用智能体运行控制台（与 Paperclip 直接对标）

### 3.1 Paperclip — 你最该研究的对标对象

- 仓库：[paperclipai/paperclip](https://github.com/paperclipai/paperclip) · 57.4k Star · MIT · TS（Node.js + React）
- 定位："The control plane for AI agents / The human control plane for AI labor"
- 最低安装：`npx paperclipai onboard --yes`，UI 默认 `localhost:3100`，要求 Node 20+
- 适配源：任何能发心跳的 Agent，列举有 OpenClaw / Claude / Cursor / Codex / Bash / HTTP

**核心 4 大视图**（这就是你应该抄/超越的对象）：

| 视图 | 作用 | 设计要点 |
|---|---|---|
| **Org Chart 组织架构图** | 可视化 Agent 的角色、汇报关系、当前状态 | 节点 = Agent；边 = 汇报关系；节点状态色（idle/working/blocked） |
| **Task Manager 工单系统** | 工单 + 线程 + 完整对话历史 | 类似 Linear 的 Issue UI，但底部接入 Agent stream |
| **Cost Dashboard 成本面板** | 实时跟踪每个 Agent 的月度预算 | 硬性预算上限（Hard Budget Limit） + 当日燃尽图 |
| **Audit Logs 审计日志** | Tool Call 全量留痕、不可变 | "What / Why / When / By Whom" 四元组 |

**它的产品哲学**（直接抄到你的 PRD）：
- "**人类是 AI 劳动力的控制平面**"——所有 Agent 永远从属于人类决策者
- 三种干预级别：Approve（批准）、Pause（暂停）、Terminate（终止）
- 强调"成本即可观测性的一等公民"——每个动作都打上 $ 标签

## 6. 赛道四：IDE/编程类 Agent（看人家怎么做"Approval UI"）

不是直接竞品，但对"**Human-in-the-Loop 干预 UI**"的设计是教科书级别——你做商业级 Agent UI 必须重点研究。

| 项目 | Star | 关键设计 |
|---|---|---|
| **Cline** | 60.5k | 每一次文件改动 / 终端命令前都要人类按 Approve；Diff 视图、命令日志、Token Counter 全部内嵌 |
| **Roo Code** | 23.3k | 在 Cline 基础上加 **Mode 切换**（Architect 计划模式 / Act 执行模式 / Ask 只问不动），是这条线最强的 UI |
| **Continue** | 32.7k | 主打"AI Code Review"，强调 `@` 引用上下文、内置 RAG |

值得抄的 3 个 UI 模式：
1. **三态指示**：Pending（待批准） / Running（运行中） / Done（已完成），用边框颜色 + 图标双重编码
2. **Diff 内嵌**：所有文件修改以"Side-by-Side Diff"+ 一键 Approve/Reject 形式呈现
3. **Cost Footer**：每条对话末尾持久显示 Token / $ / 模型；Roo 还细化到"本任务累计"

  - **Progress**：Plan、Progress Tracker

## 9. 商业项目自研路线建议

### 9.1 整体技术栈推荐

```
┌─────────────────────────────────────────────────────────┐
│  浏览器                                                  │
│  ┌────────────────────────────────────────────────────┐ │
│  │  Next.js 14 (App Router) + TypeScript              │ │
│  │  ├── UI 基座：shadcn/ui + assistant-ui + tool-ui   │ │
│  │  ├── 数据：TanStack Query + SWR                    │ │
│  │  ├── 状态：Zustand                                 │ │
│  │  ├── 实时：Server-Sent Events / WebSocket          │ │
│  │  └── 编辑器：Monaco（代码） / Tiptap（富文本）     │ │
│  └────────────────────────────────────────────────────┘ │
└────────────────────────┬────────────────────────────────┘
                         │ tRPC / REST
┌────────────────────────▼────────────────────────────────┐
│  网关层（FastAPI / NestJS）                              │
│  ├── Auth（Clerk / WorkOS / 自建）                       │
│  ├── 限流 / 审计                                         │
│  └── SSE/WebSocket Bridge                                │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│  Agent 运行时（Python，多选一或多个）                    │
│  ├── LangGraph（最主流）                                 │
│  ├── AutoGen / AgentScope（多 Agent）                    │
│  ├── CrewAI（角色式）                                    │
│  └── 自研 + MCP（最大灵活度）                            │
│                                                          │
│  沙箱：Daytona / E2B / 自建 Docker Pool                  │
│  浏览器：browser-use / Playwright                        │
│  可观测：OpenTelemetry → Langfuse / Phoenix              │
└──────────────────────────────────────────────────────────┘
```

### 9.2 UI 信息架构（直接抄这一版）

```
顶部导航
├── Workspaces（多公司/多项目）
└── 通知中心 / 用户菜单

左侧栏
├── 概览（Dashboard）
├── Agents（Org Chart 视图 ←抄 Paperclip）
├── Tasks（工单 ← 抄 Paperclip + Linear）
├── Sessions / Runs（运行历史）
├── Workflows（节点画布 ← 抄 Flowise）
├── Knowledge（知识库 ← 抄 Dify）
├── Tools / MCP（工具市场）
├── Observability（Trace、Cost、Latency ← 抄 Langfuse）
└── Settings

主区（按当前页面变化，但核心两种布局）
├── 「运行视图」= 左 Chat + 右 Workspace（Browser/Term/File/Editor，可标签切换 ← 抄 Suna/OpenHands）
└── 「编排视图」= 节点画布 + 右抽屉属性面板（← 抄 Dify/Langflow/Flowise）

底部状态栏
└── 当前会话 Token / Cost / Latency / 模型（← 抄 Cline）
```

### 9.3 Human-in-the-Loop 关键交互（商业级必做）

1. **三态徽章**：Pending（黄）/ Running（蓝脉冲）/ Done（绿）/ Failed（红） — 全平台统一色板
2. **批准卡片**：所有「写文件 / 执行命令 / 调外部 API」前出 Approval Card（参考 tool-ui 的 Approval Card）
3. **回滚到检查点**：参考 LangGraph 的 Time Travel — 每一步存 checkpoint，UI 给「跳到此步」按钮
4. **预算红线**：参考 Paperclip — 每个 Agent / 每个 Workspace 有 Hard Budget，触线自动停
5. **审计日志**：参考 Paperclip — 每条 Tool Call 不可变写入，UI 提供"Why（系统提示词 + 上下文窗口）"展开

### 9.4 License 红线（商业必读）

| 类别 | 安全 ✅ | 注意 ⚠️ | 避坑 ❌ |
|---|---|---|---|
| 直接拿来改 | MIT、Apache 2.0 | Dify/Flowise/FastGPT 等 Apache+附加条款（限制多租户 SaaS / 去 LOGO） | AGPL-3.0、GPL-3.0（AgentGPT GPL-3.0，会污染整个产品） |
| 嵌入/引用 | MIT、Apache 2.0、BSD | n8n（fair-code，禁止商业再分发）、Lobe Chat（自定义 License） | 任何"非商业"条款 |
| 论文/框架 | — | AutoGen 用 CC-BY-4.0 不是软件许可，商用须独立评估 | — |

**结论**：建议核心栈选用 `assistant-ui (MIT)` + `tool-ui (MIT)` + 自研业务层；编排画布如需要，参考 Flowise 重写或 fork 一份（注意 Flowise 自定义条款）；可观测可直接接入 Langfuse 自托管（核心 MIT）。

### 9.5 差异化机会（避免做又一个 Dify）

1. **真正的 Org Chart 化运营 UI**：Paperclip 之外几乎没人做"AI Workforce 组织管理"这件事
2. **Tool Call 可视化的"半结构化交互"**：tool-ui 才刚开始，25 个组件远不够，做"行业领域专用组件库"是大蓝海
3. **回放 + 复盘**：把 Trace 做成"会议录屏"那样可拖动的时间线（目前 Langfuse/Phoenix 还偏开发者视角）
4. **多模态 Workspace**：Suna/OpenHands 只做了浏览器+终端，加入"实时白板 / 表格 / Slides"是机会
5. **预算与 SLA**：成本看板做到分项目 / 分 Agent / 分客户三层，是 To B 必杀
